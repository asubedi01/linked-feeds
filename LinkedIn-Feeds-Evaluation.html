LinkedIn Feeds: What the API Will and Won't Let Us Build

Can we ship an Instagram-Feeds-style product for LinkedIn — user connects, we fetch their posts, posts display on their website? This is the result of live API testing against a real LinkedIn developer app.

Date: June 16, 2026 App tested: ClickSocial (client ID 783vnel7uw3ggw) Method: Live OAuth + API probes, two real admin Pages Status: Evaluation complete

Bottom line

The blocker here is not technical capability — it's LinkedIn's terms of service. Unlike Instagram, Facebook, or TikTok, LinkedIn has explicitly named our product category as a prohibited use case. We confirmed this with live API calls, not just by reading docs.

Not feasible

Personal member feeds

Reading a member's own posts requires r_member_social — a permission LinkedIn has closed to all new applicants. No other scope substitutes. Confirmed dead via live testing.

Technically works, contractually banned

Company / organization Page feeds

The API does return full company-page post content (we verified 200 responses). But LinkedIn's "No Social Feeds" policy explicitly bans displaying this on a website.

The exact prohibition — verbatim from LinkedIn's policy:
"No Social Feeds: Under our Marketing API Terms, none of the data provided via our Community Management APIs can be used in a social feed use case (e.g. to display a feed of LinkedIn company updates on the company's website or intranet)." Restricted Uses of LinkedIn Marketing APIs and Data — verified live, June 2026

Violating it risks API access revocation, which on a shared developer app would also take down ClickSocial's production posting integration. So a fetch-and-display feed product is off the table on this app. The viable paths (below) avoid the read API entirely.

Scopes & capabilities — what we actually have

The app's OAuth consent returns this exact scope string:

r_basicprofile, r_member_postAnalytics, r_organization_social, r_organization_social_feed, rw_organization_admin

r_member_social is the crux. LinkedIn's docs state they are "not accepting access requests at this time due to resource constraints," and it's absent from the permission table even approved Community Management partners receive. There is no path to it today.

The limitation stack

Even setting aside the headline ban, a fetch-and-display feed runs into a wall of independent blockers — ordered by severity. Any one of the top four is sufficient to stop the product.

  1. "No Social Feeds" restricted useThe product category itself is banned for Community Management API data. Violation cost: API revocation, which also takes down ClickSocial's production posting.
  2. 48-hour storage capMember social activity data can't be cached beyond 48 hours — directly conflicts with a cache-in-WordPress-database architecture.
  3. Limited Audience ruleData obtained to manage a Page may only be shown "to individuals associated with that Page." A public website visitor doesn't qualify — this alone blocks public display.
  4. r_member_social is closedPersonal feeds are dead regardless of everything else.
  5. Standard-tier gateDev tier caps at 500 calls/app/day, 100/member/day. Upgrade requires a manual review with a screencast of the use case — which would show LinkedIn the exact product they prohibit.
  6. Token lifecycle60-day access token, 365-day refresh token. Programmatic refresh works, but annual interactive re-auth per customer is the hard floor.
  7. Versioning taxMonthly API versions, sunset at ~12 months. An annual-minimum maintenance commitment per shipped plugin.
  8. Unpublished rate limitsPer-endpoint limits are only visible in the dev portal after making calls — capacity planning is empirical.

Ways forward — what we could ship

The viable options sidestep the read API and the social-feed ban entirely. In descending order of compliance confidence:

Publish-and-display loop Strongest wedge

A plugin that publishes to LinkedIn from WordPress (the write scopes w_member_social / w_organization_social are unrestricted, including for personal profiles). Since the post originates in the customer's own WP composer, the site already holds the content and the post URN — it can render an "our LinkedIn posts" wall from its own local copies, with deep links to the live posts. No read API call, no Community-Management data displayed.

Why it's defensible: the ban covers "data provided via our Community Management APIs." Customer-authored content originating on their own site isn't that. Covers both personal and company posts — the only route that reaches personal profiles. Limitation: shows only posts made through the plugin, not pre-existing native posts. Needs a legal sanity check.

Curated-embeds plugin ToS-clean

LinkedIn officially supports single-post iframe embeds. A plugin where users paste post URLs and we render a styled grid of official embeds works for both personal and company posts — no API, no auth, no data storage.

Limitation: manual curation (no auto-discovery — that needs the closed read API), and iframe styling constraints. Honest framing: a "LinkedIn post showcase," not a live feed. Low build cost; good way to validate category demand.

Paths that are not viable

Fetch-and-display company feed — technically works, but banned by "No Social Feeds" + the 48-hour storage cap + the Limited Audience rule. Scraping (how some competitors offer personal-profile feeds without OAuth) — a worsening legal bet: LinkedIn sued Proxycurl in Jan 2025; it shut down July 2025; Apollo.io and Seamless.AI lost their LinkedIn pages in March 2025. A non-starter for Smash Balloon's brand and WordPress.org standing. Member Data Portability (DMA) — EEA members only, separate purpose-restricted terms; not a global feed foundation.

Recommendation

LinkedIn is structurally different from the other networks Smash Balloon supports: the gap is legal, not technical, and the one scope that would enable a true personal feed is closed with no path to it.

Don't pursue a fetch-and-display feed on a shared production app — the revocation risk to ClickSocial's posting integration is real and the use case is explicitly banned.

If we want a LinkedIn entry in the catalog, the publish-and-display loop is the strongest, most defensible "feeds" wedge and the only one that reaches personal profiles; the curated-embeds plugin is the lowest-cost way to test demand for the category. Both warrant a brief legal review before build.
